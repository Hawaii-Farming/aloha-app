<objective>
Fix the module-row navigation regression discovered via E2E. The actual root
cause is that `app/routes/workspace/module.tsx:24` queries a view
(`app_nav_sub_modules`) that does not exist in the hosted database. PostgREST
returns 404 (PGRST205); castRows on the error result returns `[]`, so the
`subModules.length > 0` branch is never taken and the loader always falls
through to `throw redirect('/home/${accountSlug}')`.

Align docs to reality: the DB has `app_navigation` (one row per accessible
sub-module with parent module info). It does NOT have `app_nav_modules` or
`app_nav_sub_modules`. Multiple docs lied about this.
</objective>

<evidence>
Probed hosted Supabase from a logged-in browser session:
- `GET /rest/v1/app_navigation?select=count&limit=1` ‚Üí 200 [{"count": 0}]
- `GET /rest/v1/app_nav_sub_modules?select=count&limit=1` ‚Üí 404 PGRST205
  "Could not find the table 'public.app_nav_sub_modules'"
- `GET /rest/v1/app_nav_modules?select=count&limit=1` ‚Üí 404 PGRST205

`module.tsx` is byte-identical between `main` and `design`, so this bug
existed on main all along. It went undetected because:
1. The @phase10 E2E test that would have caught it was waived at Phase 10
   close (unrelated setup bugs, fixed in 260416-ppa).
2. Plan 10-05 "BUG-01 fix" only verified click-handling on sidebar
   module buttons (manual smoke). Nobody tested whether the module
   landing route actually resolved to a sub-module.
3. The sidebar uses a different code path (`loadOrgWorkspace` ‚Üí
   `app_navigation`) which works, so the HR link renders with a valid-
   looking href.
</evidence>

Replace the query block that targets `app_nav_sub_modules`:

```ts
const { data } = await queryUntypedView(client, 'app_navigation')
  .select('sub_module_slug, sub_module_display_order')
  .eq('org_id', accountSlug)
  .eq('module_slug', moduleSlug)
  .order('sub_module_display_order');
```

Column names in `app_navigation` are `sub_module_display_name` and
`sub_module_display_order` (not `display_name` / `display_order`), so the
order clause and select list need those names. `castRows<{ sub_module_slug }>`
stays ‚Äî we only read `sub_module_slug[0]` to build the redirect URL.

Leave the `throw redirect` branches unchanged ‚Äî they're correct. The query
fix is what makes `subModules.length > 0` actually true for modules that
have accessible sub-modules.
